## Permission Actions

Each resource supports the following actions (where applicable):

- **create** - Create new records
- **read** - View and read records
- **update** - Modify existing records
- **delete** - Remove records
- **approve** - Approve requests or transactions
- **reject** - Reject requests
- **export** - Export data to files
- **import** - Import data from files
- **assign** - Assign tasks or resources
- **manage** - Full management access
- **view_all** - View all records (not just own)
- **view_own** - View only own records
- **edit_all** - Edit all records
- **edit_own** - Edit only own records
- **cancel** - Cancel operations
- **refund** - Process refunds
- **void** - Void transactions
- **print** - Print documents

## Database Structure

### Core Tables

**roles**
- Stores all role definitions
- Fields: id, name, scope, is_system_role, created_at

**permissions**
- Defines all possible system permissions
- Fields: id, resource, action, description, category, is_system_permission, created_at

**user_role_branch**
- Maps users to roles at specific branches
- Fields: id, user_id, role_id, branch_id, is_primary_role, assigned_at

**role_permissions**
- Maps permissions to roles
- Fields: id, role_id, permission_id, granted, created_at

**permission_overrides**
- User-specific permission overrides
- Fields: id, garage_id, user_id, resource, action, allowed, reason, created_by, expires_at, created_at

## Usage Examples

### Assigning Roles to Users

```typescript
// Assign role to user at a branch
await db.insert(userRoleBranch).values({
  userId: "user-123",
  roleId: "role-456",
  branchId: "branch-789",
  isPrimaryRole: true,
});
```

### Checking User Permissions

```typescript
// Check if user has permission
import { hasPermission } from './rbac-middleware';

const canCreateInvoice = await hasPermission(
  userId,
  'invoices',
  'create',
  garageId
);
```

### Role-Based Route Protection

```typescript
// Protect route with role check
app.get('/admin/users', 
  requireRole(['System Administrator', 'HR Manager']),
  async (req, res) => {
    // Handler code
  }
);
```

## Best Practices

### Security
- Always use role-based checks on both frontend and backend
- Never trust client-side role checks alone
- Implement permission overrides carefully with audit trails
- Use temporary permissions with expiration dates for sensitive access

### Performance
- Cache user permissions in session
- Minimize database queries with eager loading
- Use permission checks at route level, not per-data-item

### Maintenance
- Review and update role permissions quarterly
- Audit permission overrides regularly
- Document custom permissions thoroughly
- Test role changes in staging before production

## Migration Guide

### Adding a New Role

1. Define role in `server/rbac-config.ts`:
```typescript
NEW_ROLE: {
  name: 'New Role Name',
  scope: 'branch', // or 'garage', 'system', 'franchise'
  isSystemRole: false,
  description: 'Role description'
}
```

2. Define permissions in `ROLE_PERMISSIONS`:
```typescript
NEW_ROLE: {
  resources: {
    customers: [ACTIONS.READ, ACTIONS.CREATE],
    invoices: [ACTIONS.VIEW_OWN],
  }
}
```

3. Run seeding script:
```bash
tsx server/seed-rbac.ts
```

### Adding a New Permission Resource

1. Add to `RESOURCES` in `server/rbac-config.ts`:
```typescript
new_feature: { 
  category: PERMISSION_CATEGORIES.DASHBOARD, 
  label: 'New Feature' 
}
```

2. Update role permissions as needed
3. Reseed permissions
